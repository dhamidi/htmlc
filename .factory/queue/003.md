# Rewrite renderComponentElement to Use Deferred Slot Definitions

## Description

Update `renderComponentElement` in `renderer.go` to collect `SlotDefinition`s instead of pre-rendering slot content to a string. This is the key architectural change that enables scoped slots: because the child component may pass data back up to the caller's slot content, the slot content cannot be rendered until the child's `<slot>` element is reached — at which point the slot props are available.

**Current behaviour to replace:** the caller's children are serialised to an HTML string and injected as the `$slot` variable into the child scope. This prevents scoped slots because the child's data does not yet exist when the string is produced.

**New behaviour:**

1. **Detect `v-slot` on the component tag itself.** Scan the component element's attributes for a `v-slot` or `#default`/`#name` directive. If found:
   - Parse the slot name and binding pattern from the attribute.
   - Collect *all* of the component element's children as the content for that slot.
   - Store a single `SlotDefinition` in the map under the parsed slot name.
   - **Validation**: if the component element also contains any `<template v-slot:...>` or `<template #...>` children, return a render error:
     `component "<Name>": v-slot on component tag cannot be mixed with named slot templates`

2. **Otherwise**, call `collectSlotDefs(componentElement, currentScope)` to build the slot map from the element's children (Task 002).

3. Build the **child scope** from the component element's non-slot attributes exactly as before:
   - Evaluate `:propName="expr"` and `v-bind:propName="expr"` bindings in the current scope.
   - Include static attributes as string values.
   - Skip slot directive attributes (they have been consumed in step 1).

4. Instantiate the **child renderer** with the child component's template, the shared registry, the style collector, and the slot definitions map.

5. Call `childRenderer.Render(w, childScope)`.

**What does not change:** the child renderer's scope is still built from the component element's props, not from the slot definitions. Slot props are merged into the *slot content's* scope at render time (Task 004).

## Acceptance Criteria

- Slot content is no longer pre-rendered to a string; the `$slot` mechanism is removed.
- The child renderer receives a `map[string]*SlotDefinition` instead of a `$slot` string.
- `v-slot="{ a, b }"` on the component tag wraps all children as the default slot with the given binding pattern.
- `#header="{ title }"` on the component tag wraps all children as the `"header"` slot with the given binding pattern.
- Mixing `v-slot` on the component tag with `<template #namedSlot>` children produces a render error containing the component name.
- A component element with no children and no slot directive produces an empty slot definitions map (no crash, fallback content renders correctly).
- All previously passing tests that exercise the default slot continue to pass (backward compatibility preserved).
