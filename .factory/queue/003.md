# Add scoped slot support

## Description

Extend the slot rendering from task 002 to support scoped slots — the child component passing data back to the caller's slot template via slot props on the `<slot>` element.

**Changes to `<slot>` element rendering:**

When a `SlotDefinition` is found for the current `<slot>` element:

1. **Collect slot props**: iterate over the `<slot>` element's attributes in the child's scope:
   - `:attr="expr"` or `v-bind:attr="expr"` → evaluate `expr` in the child's current scope, add to slot props map as `attr → value`
   - Static attributes (other than `name`) → add as string values to slot props map
   - The `name` attribute is excluded from slot props
2. **Build render scope**: clone the slot definition's `ParentScope`, then apply the binding pattern:
   - No binding (`BindingVar` and `Bindings` both empty): render scope = cloned parent scope only, slot props are discarded
   - Single variable (`BindingVar = "props"`): set `renderScope["props"] = slotPropsMap`
   - Destructured (`Bindings = ["user", "index"]`): for each variable name, set `renderScope[name] = slotPropsMap[name]`; missing keys become `nil`
3. **Render** the slot definition's `Nodes` with the merged render scope.

**Scope rules:**
- Slot props override parent scope variables if names conflict (within the slot content only)
- Slot props do not leak into the parent's scope outside the slot content
- The clone ensures the parent scope is not mutated

## Acceptance Criteria

- `<slot>` element dynamic bindings (`:attr="expr"`) are evaluated in the child's current scope and collected as slot props
- Static attributes on `<slot>` (other than `name`) are included as string slot props
- The `name` attribute is not included in slot props
- When a slot definition has a single variable binding, the entire slot props map is bound to that variable name
- When a slot definition has destructured bindings, each named variable receives its corresponding value from slot props; missing keys are `nil`
- When a slot definition has no binding, slot props are not merged into the render scope
- Parent scope variables are accessible in slot content
- Slot props override parent scope variables of the same name (within slot content only)
- Scoped slot works inside `v-for` — each iteration passes different slot prop values
- New tests in `renderer_test.go`:
  - Child passes `:user="user"` on `<slot>`, caller receives via `#default="{ user }"`
  - Single variable binding: `v-slot="slotProps"` receives the full props object, accessible as `slotProps.msg`
  - Scoped slot inside `v-for`: each iteration provides different data to the slot
  - Slot content accesses both parent scope variables and slot props
  - Slot prop overrides parent variable of same name within slot content
  - Named scoped slot: `<slot name="item" :user="user" />` with `<template #item="{ user }">`
- All existing tests continue to pass
