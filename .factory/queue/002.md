# expr: write explanation documentation

## Description

Add an explanation section to the `expr` package that covers the _why_ and _how_ behind its design. The explanation quadrant of Diátaxis is understanding-oriented: it illuminates a topic, discusses trade-offs, and connects concepts. It answers "why does it work this way?" rather than "how do I use it?" or "what does X mean?".

The explanation must be added as a clearly delimited section inside the `doc.go` file created in the reference task (or as a separate `explanation.go` file with only a package clause and doc comment if that is cleaner).

Topics to cover:

1. **Purpose and scope**: The `expr` package implements a small subset of JavaScript expression semantics sufficient for Vue.js template attribute bindings (`:attr="expr"`, `{{ interpolation }}`, `v-if="expr"`, etc.). It is intentionally not a general-purpose JS engine.

2. **Pipeline architecture**: Source text → `Lexer` (tokens) → `Parser` (AST) → `Eval` (value). Each stage is separate; explain why (testability, reuse by `identifiers.go` which only needs the AST stage, not evaluation).

3. **JavaScript-like semantics in Go**: Explain the deliberate mapping — e.g. all numbers are `float64` (matching JS), `null` maps to Go `nil`, `undefined` is a sentinel struct rather than a zero value, truthy/falsy rules follow JS (empty string and `0` are falsy).

4. **Scope as `map[string]any`**: Why the scope is a plain Go map rather than a struct or interface. Discuss the trade-off: flexibility for the template engine vs. loss of compile-time checking.

5. **`UndefinedValue` vs `nil`**: Why two absent-value sentinels exist. `nil` (Go/JSON `null`) represents an intentionally-null value. `UndefinedValue` represents a missing key — the difference matters for truthiness and equality checks.

6. **Why a subset, not full JS**: Justify the unsupported constructs (no assignment, no closures, no `new`) — keeping the evaluator stateless makes templates predictable and safe for server-side rendering.

## Acceptance Criteria

- The explanation text is present in `expr/doc.go` (or a separate `expr/explanation.go`) as a Go doc comment
- All six topics listed above are addressed
- The text does not contain step-by-step instructions or syntax tables (those belong in the reference)
- The explanation uses the word "because" or equivalent causal language at least once per topic, reflecting that it is answering _why_ questions
- The file compiles (`go build ./expr/` passes)
- `go doc ./expr/` output includes the explanation text
