# Integrate with stdlib fs.FS interface

## Description

The `Engine` currently reads `.vue` files directly from the OS filesystem via
`os.ReadFile` and `filepath.WalkDir`. This prevents callers from using embedded
filesystems (`//go:embed`), in-memory virtual filesystems, or any other
`fs.FS` implementation.

Add first-class support for `fs.FS` in the Engine API so that callers can,
for example, embed their component tree into the binary:

```go
//go:embed templates
var templateFS embed.FS

engine, err := htmlc.New(htmlc.Options{
    FS:           templateFS,
    ComponentDir: "templates",
})
```

### Changes required

#### `engine.go`

1. Add `FS fs.FS` field to the `Options` struct with a doc comment explaining
   that when set it is used instead of the OS filesystem for all file reads and
   directory walks. `ComponentDir` is then interpreted as a path within the FS.

2. Update `discover(dir string)` to branch on whether `opts.FS` is set:
   - When set: use `fs.WalkDir(opts.FS, dir, ﻗ۵)` and read file contents with
     `fs.ReadFile(opts.FS, path)`.
   - When not set: keep the existing `filepath.WalkDir` / `os.ReadFile` path.

3. Update `registerPath(name, path string)` similarly: use
   `fs.ReadFile(opts.FS, path)` when `opts.FS != nil`, otherwise `os.ReadFile`.

4. Update `maybeReload()` hot-reload support:
   - When `opts.FS` is not nil, check whether the FS also satisfies `fs.StatFS`.
     If it does, use `fsys.Stat(path)` to obtain the mtime and compare as
     before. If it does not implement `fs.StatFS`, skip reload for that entry
     (or disable reload entirely with a warning in the `New` constructor when
     both `FS` and `Reload` are set).

5. Update `Register(name, path string)` to use the FS when set, so manually
   registered components can also be read from the embedded FS.

#### `engine_test.go`

Add tests using `testing/fstest.MapFS` (from the stdlib) as a lightweight
in-memory FS:
- `TestNew_WithFS_DiscoverAndRender` ﻗ create a `MapFS` with two `.vue` files,
  build an engine with `FS` set, and verify both components render correctly.
- `TestNew_WithFS_ComponentDir` ﻗ verify that `ComponentDir` is interpreted as
  a sub-path within the FS.
- `TestNew_WithFS_NoReload` ﻗ verify that when `FS` is set and the FS does not
  implement `fs.StatFS`, hot-reload does not crash (it is silently skipped).
- `TestEngine_Register_WithFS` ﻗ verify that `Register` reads from the FS when
  one is configured.

### Constraints

- When `opts.FS` is nil the behaviour must be identical to today; no existing
  tests should change.
- The `FS` field must accept any `fs.FS` implementation ﻗ do not type-assert to
  `embed.FS` or any concrete type.
- Import `"io/fs"` in `engine.go`; do not add any other new dependencies.

## Acceptance Criteria

- `Options` has an `FS fs.FS` field.
- When `Options.FS` is non-nil, `New` reads and walks component files through
  that FS rather than the OS filesystem.
- When `Options.FS` is nil, behaviour is unchanged (all existing engine tests
  still pass).
- `TestNew_WithFS_DiscoverAndRender` passes: components loaded from a
  `testing/fstest.MapFS` render the expected HTML.
- `TestNew_WithFS_NoReload` passes: setting `Reload: true` with a non-`StatFS`
  FS does not panic or error.
- `go test ./...` passes with no new failures.
- README documents the embedded FS pattern.
