# Refactor slot collection and rendering for named slots

## Description

Replace the pre-rendered `$slot` string mechanism with `SlotDefinition`-based slot collection and rendering. This is the core change that enables named slots.

**Changes to `renderComponentElement` (slot collection):**

When processing a component invocation, instead of pre-rendering all children to a string and injecting `$slot`:

1. Scan children for `<template>` elements with `v-slot:name` or `#name` attributes (using `parseSlotDirective` from task 001).
2. Each matching `<template>` creates a named `SlotDefinition` with:
   - `Nodes`: the `<template>`'s children (not the `<template>` itself)
   - `ParentScope`: a clone of the caller's current scope
   - Binding info from `parseBindingPattern` on the attribute value
3. All remaining children (not wrapped in `<template v-slot:...>`) form the default slot's `SlotDefinition`.
4. Pass the `map[string]*SlotDefinition` to the child renderer (add a `slotDefs` field to the `Renderer` struct, or pass via a well-known scope key like `$slotDefs`).

**Changes to `<slot>` element rendering:**

When the renderer encounters a `<slot>` element:

1. Read the `name` attribute (default: `"default"`).
2. Look up the `SlotDefinition` for that name.
3. If **not found**: render the `<slot>` element's own children as fallback content (in the child's scope) — this preserves existing fallback behavior.
4. If **found**: render the slot definition's `Nodes` using the slot definition's `ParentScope` (for now, without scoped slot props — that comes in task 003).

**Backward compatibility:**

- Remove the `$slot` string injection from `renderComponentElement`.
- Remove the `$slot` string check from `<slot>` element rendering.
- Plain `<MyComponent><p>hello</p></MyComponent>` must continue working: children with no `v-slot` directives become the default slot definition, rendered when `<slot />` is encountered.
- Fallback content (`<slot>Default text</slot>`) must continue working when no content is provided.

## Acceptance Criteria

- `renderComponentElement` no longer pre-renders children to a `$slot` string
- `renderComponentElement` collects `<template v-slot:name>` / `<template #name>` children into named `SlotDefinition` entries
- Remaining children (not in `<template v-slot:...>`) are collected as the default slot definition
- The `<slot>` element handler reads the `name` attribute and looks up the corresponding `SlotDefinition`
- If a slot definition exists, its `Nodes` are rendered using its `ParentScope`
- If no slot definition exists, the `<slot>` element's children are rendered as fallback (in child scope)
- The `<slot>` element itself produces no wrapper tag in the output
- All existing tests pass without modification (backward compatibility with plain slot usage)
- New tests in `renderer_test.go`:
  - Named slots: component with `<slot name="header" />` and `<slot name="footer" />` receives content via `<template #header>` and `<template #footer>`
  - Default slot content coexists with named slots
  - Missing named slot renders fallback content
  - Provided named slot overrides fallback content
