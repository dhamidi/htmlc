# expr: write tutorial documentation

## Description

Add a tutorial for the `expr` package as a `tutorial_test.go` file in `expr/`. The tutorial quadrant of Diátaxis is learning-oriented: it takes a complete beginner through a series of steps and teaches by doing. Its primary purpose is to build confidence and establish a mental model, not to be a comprehensive reference.

The tutorial should assume the reader is a Go developer who has never used the `expr` package before. It guides them from the simplest possible expression through progressively more interesting concepts, in a single narrative test function `TestTutorial`.

**Structure** (each step is a sub-section with a prose comment explaining what is being learned):

1. **Evaluate a number literal** — call `expr.Parse("42")` and `Eval(nil)`, observe `float64(42)`. Teaches: Parse returns an `Expr`, Eval takes a scope.

2. **Evaluate an identifier from scope** — `expr.Parse("name")` with scope `{"name": "Alice"}`. Teaches: scope is a `map[string]any`; identifiers resolve against it.

3. **Use arithmetic** — `expr.Parse("price * quantity")` with a scope containing both values. Teaches: binary operators work on scope variables.

4. **Access a nested field** — `expr.Parse("user.name")` with a nested map. Teaches: dot notation traverses maps.

5. **Use a conditional expression** — `expr.Parse("stock > 0 ? \"In stock\" : \"Out of stock\"")`. Teaches: ternary; string literals.

6. **Handle a missing key** — evaluate `expr.Parse("missing")` against an empty scope; show the result is `expr.UndefinedValue{}`, not an error. Teaches: missing keys are not errors; the distinction between `undefined` and `null`.

7. **Discover what a expression depends on** — call `expr.CollectIdentifiers("price * quantity + tax")` and show the returned set. Teaches: identifier collection for prop discovery.

The tutorial must:
- Be a single `TestTutorial` function using `t.Run` sub-tests for each step
- Use `testing` standard library only (no external assertion libraries)
- Include prose comments above each sub-test explaining what the reader is learning and why it matters
- Avoid error-handling boilerplate that distracts from the lesson — use `t.Fatalf` immediately on any unexpected error

## Acceptance Criteria

- `expr/tutorial_test.go` exists with `TestTutorial` containing all seven steps as sub-tests
- `go test ./expr/ -run TestTutorial` passes
- Each sub-test has a prose comment of at least one sentence explaining what concept is being taught
- The tutorial reads as a narrative progression — each step builds on the previous one
- No step duplicates content from the how-to examples in `example_test.go`
