# Implement Scoped <slot> Element Rendering with Scope Merging

## Description

Rewrite the `<slot>` element handling inside `renderElement` in `renderer.go` to use the `SlotDefinition` map provided by the parent, evaluate slot props, merge scopes according to the binding pattern, and render the caller's AST nodes in the resulting scope.

**Rendering algorithm for a `<slot>` element:**

1. **Determine slot name:** read the element's `name` attribute (static string). Default: `"default"`. The `<slot>` element itself is never written to the output.

2. **Look up the SlotDefinition** for that name in the renderer's slot definitions map.

3. **No definition found → render fallback content:** render the `<slot>` element's own children in the *child component's current scope* (the scope in effect when the `<slot>` element is reached). This is the standard fallback / default content behaviour.

4. **Definition found → render caller's content with merged scope:**

   a. **Collect slot props:** iterate over all attributes of the `<slot>` element. For each attribute:
      - If the attribute name starts with `:` or `v-bind:`, evaluate the attribute value expression in the **child's current scope** and add the result to the slot props map under the attribute name with the prefix stripped.
      - Static attributes (other than `name`) are added as their raw string value.
      - The `name` attribute itself is excluded from slot props.

   b. **Build the render scope:** start with a shallow clone of `SlotDefinition.ParentScope`, then apply the binding pattern:
      - *No binding* (`BindingVar == ""` and `Bindings == nil`): the render scope is the parent scope clone as-is.
      - *Single variable* (`BindingVar != ""`): set `renderScope[BindingVar] = slotPropsMap`.
      - *Destructured* (`Bindings != nil`): for each name in `Bindings`, set `renderScope[name] = slotPropsMap[name]` (missing keys become `nil`).

   c. **Render** each node in `SlotDefinition.Nodes` into the output using the render scope. Use the same `renderNode` / `renderElement` machinery already used elsewhere.

**Scope isolation:** slot props must not leak into the parent's scope outside the slot content. Because the render scope is built from a *clone* of `ParentScope`, the original caller's scope remains unchanged after the slot content is rendered.

**`<template>` transparency:** if the `<slot>` element appears inside a `<template v-if>` or `<template v-for>` block, it is subject to normal conditional/loop evaluation first; the slot-specific logic above applies only when the `<slot>` element is actually being rendered.

## Acceptance Criteria

- A `<slot />` with no corresponding caller definition renders the slot's fallback children in the child's scope.
- A `<slot />` with a matching definition renders the caller's nodes; fallback children are ignored.
- A `<slot name="header" />` renders the caller's `#header` content; an unmatched named slot renders its own fallback children.
- Static attributes on `<slot static-prop="value" />` are included in slot props as strings (excluding `name`).
- Dynamic attributes `:msg="greeting"` on `<slot>` are evaluated in the child's scope and included in slot props.
- `v-slot="{ user, index }"` on the caller side destructures slot props: `user` and `index` are individually available in the slot content.
- `v-slot="slotProps"` on the caller side binds the entire slot props object to `slotProps`.
- A slot with no binding pattern (`v-slot` or `#name` with no `=`) still receives slot props but ignores them (no crash).
- Slot props override same-named parent scope variables within the slot content.
- Slot props do not appear in the parent's scope outside the slot content.
- Extra slot prop keys not named in a destructure pattern are silently ignored.
- Scoped slots work correctly inside `v-for` loops (each iteration provides independent slot props).
- The `<slot>` element itself produces no output in the final HTML.
