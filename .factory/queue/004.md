# Support v-slot on component tag and add mixed usage validation

## Description

Add support for `v-slot` or `#default` appearing directly on the component element tag (rather than on a `<template>` child). This form means "all my children are the default slot content" and is the idiomatic way to use scoped slots when only a default slot is needed.

Also add validation to prevent mixing `v-slot` on the component tag with `<template v-slot:...>` children, which is invalid per Vue's rules.

**Changes to `renderComponentElement` (slot collection):**

Before scanning children for `<template v-slot:...>`:

1. Check the component element's own attributes for `v-slot`, `v-slot:default`, `#default`, or any `v-slot:name` / `#name` attribute (using `parseSlotDirective`).
2. If found on the component tag:
   a. Parse the binding pattern from the attribute value.
   b. Check if any children are `<template>` elements with slot directives. If so, return a render error: `component "ComponentName": v-slot on component tag cannot be mixed with named slot templates`
   c. Collect all children as the default slot definition with the parsed binding pattern and the caller's scope.
3. If not found on the component tag, proceed with the existing child-scanning logic from task 002.

**Shorthand equivalence:**
- `v-slot="{ item }"` on component tag = `#default="{ item }"` on component tag
- Both mean: all children are default slot content, with destructured binding

## Acceptance Criteria

- `v-slot="{ item }"` on a component tag collects all children as the default slot with the given binding pattern
- `#default="{ item }"` on a component tag works identically
- `v-slot` (no value) on a component tag collects all children as default slot with no binding
- When `v-slot` is on the component tag, slot props from the child's `<slot />` are merged per the binding pattern
- Mixing `v-slot` on component tag with `<template #named>` children produces a render error with message: `component "ComponentName": v-slot on component tag cannot be mixed with named slot templates`
- New tests in `renderer_test.go`:
  - `v-slot="{ user, index }"` on component tag with scoped slot works (the main scoped slot example from the spec)
  - `#default="{ item }"` on component tag works
  - `v-slot` (no value, no binding) on component tag works
  - Mixed usage error: component tag has `v-slot` and children include `<template #header>` → error message matches expected format
- All existing tests continue to pass
