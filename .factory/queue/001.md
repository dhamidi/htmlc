# expr: write reference documentation

## Description

Add a complete reference document for the `expr` expression language as a `doc.go` file in the `expr/` package. The reference quadrant of Diátaxis covers information-oriented material: accurate, complete, and to the point. It describes the machinery — not how to use it or why it works the way it does.

The reference must cover:

- **Literals**: number (integer and float), string (single- and double-quoted), boolean (`true`/`false`), `null`, `undefined`
- **Identifiers**: how names are resolved against the scope map
- **Operators** (with precedence table, highest to lowest):
  - Unary: `!`, `-`, `+`
  - Multiplicative: `*`, `/`, `%`
  - Additive: `+`, `-`
  - Relational: `<`, `>`, `<=`, `>=`
  - Equality: `==`, `!=`
  - Logical AND: `&&`
  - Logical OR: `||`
  - Ternary: `condition ? then : else`
- **Member access**: dot notation (`a.b`) and bracket notation (`a["b"]`, `a[0]`)
- **Function calls**: `f(arg1, arg2)`
- **Built-in functions**: `len(value)` — returns the length of a string, slice, array, or map as `float64`
- **Type coercion rules**: how Go values are compared and coerced (e.g. numeric comparisons always use `float64`)
- **`UndefinedValue`**: the sentinel type returned when a key is absent from the scope; distinct from `nil`/`null`
- **Unsupported constructs** (things that look like JS but are not supported): assignment, `typeof`, `instanceof`, template literals, spread, arrow functions, `new`

The document must **not** include usage tutorials, how-to recipes, or design rationale. Pure reference.

## Acceptance Criteria

- A `doc.go` file exists in `expr/` containing a package-level doc comment that is recognisable as a Diátaxis reference: it is organised around what the language _is_ (syntax and semantics) rather than what to _do_ with it
- Every operator listed in the spec above appears with its precedence level
- `UndefinedValue` is documented as distinct from `nil`
- The built-in `len()` function is documented with its argument type, return type, and error condition
- Unsupported constructs are explicitly listed so callers know not to rely on them
- The file compiles (`go build ./expr/` passes)
- No tutorial prose, step-by-step instructions, or "how to" framing appears in the document
