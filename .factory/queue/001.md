# Improve package documentation so go doc -all reads well

## Description

`go doc -all github.com/dhamidi/htmlc` currently shows four separate package-level
doc comments (one from each source file). The individual symbol comments are mostly
one-liners that restate the name. There are no examples. The documentation does not
explain what htmlc is, why it exists, or how the pieces fit together.

Rewrite the documentation so that `go doc -all` follows the Diátaxis framework:
- **Explanation**: what htmlc is, the problem it solves, the mental model
- **Reference**: every exported symbol clearly described
- **Tutorial**: at least one working Example function

### Concrete changes

1. **Create `doc.go`** with a single package-level doc comment that:
   - Opens with one sentence: what htmlc is and the problem it solves
   - Describes the four-concept model: Engine → Component (parsed .vue file) →
     Renderer → StyleCollector
   - Shows a minimal code snippet (as a doc comment `//   engine, _ := htmlc.New(...)`)
     illustrating typical use
   - Replaces all per-file "Package htmlc …" sentences (remove them from component.go,
     renderer.go, style.go, engine.go so there is only one package doc comment)

2. **Improve type and function doc comments** (reference):
   - `Engine`: "Engine is the entry point for rendering .vue components. Create one
     with New; call RenderPage or RenderFragment to produce HTML."
   - `Options.ComponentDir`: explain that components are discovered recursively and
     registered by basename without extension
   - `Options.Reload`: explain the dev-mode use case (hot reload on mtime change)
   - `RenderPage` vs `RenderFragment`: the difference must be clear — RenderPage
     injects styles before `</head>` (for full HTML documents); RenderFragment
     prepends styles (for partial updates / HTMX)
   - `Renderer` and `Registry`: note they are the low-level API; most callers use
     Engine instead

3. **Add `example_test.go`** with at least one `Example` function showing end-to-end
   use: `htmlc.New` → write a temp .vue file → `engine.RenderPage` → print output.
   The example must compile and pass `go test` (use `// Output:` comment or
   `// Unordered output:` if order varies).

## Acceptance Criteria

- `go doc -all github.com/dhamidi/htmlc` shows exactly **one** package-level doc
  comment (not four)
- The package comment is at least 5 lines and mentions: "Vue", "server-side" (or
  "server side"), "scoped styles", and "net/http" (or "http.Handler")
- Every exported symbol has a non-trivial doc comment (not just the symbol name
  restated)
- `RenderPage` and `RenderFragment` doc comments explicitly state how they differ
- `go doc -all` output includes at least one `Example` function
- `go test ./...` continues to pass with no failures
