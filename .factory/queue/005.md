# Update Props() to Exclude v-slot Binding Variables

## Description

Update `walkForProps` in `component.go` so that variables introduced by `v-slot` binding patterns are treated as **locally scoped** identifiers — the same way `v-for` loop variables are handled today. Without this change, the `Props()` method incorrectly reports slot-bound variables (e.g. `user` and `index` from `v-slot="{ user, index }"`) as external props that must be supplied by the caller.

**Changes to `walkForProps`:**

When processing an element's attributes, check each attribute name with `parseSlotDirective`. If the attribute is a slot directive:

1. Parse the directive's value with `parseBindingPattern` to extract the binding variable(s).
2. Add those variable names to the *local scope clone* for the element's children (mirror the existing logic for `v-for` loop variables).
3. Do **not** add the directive value's contents to the parent scope's prop collection (the binding pattern is not an expression that references external data).

**The `#` shorthand must be handled.** HTML parsers lowercase attribute names, so `#header` is already lowercase-safe, but `v-slot:Header` would be lowercased to `v-slot:header`. The `parseSlotDirective` helper from Task 001 handles both forms, so calling it here is sufficient.

**What does not change:**

- Dynamic bindings (`:prop="expr"`) on a `<slot>` element inside the *component's own template* are expressions evaluated in the component's scope. Identifiers in those expressions correctly remain as candidate props (they are the child component's own data, not caller-supplied).
- All existing `v-for` variable scoping logic is unchanged.

## Acceptance Criteria

- `Props()` on a component whose template contains `<slot :data="items" />` includes `items` as a prop (it is referenced by the component itself).
- `Props()` on a component whose template uses `<MyComp v-slot="{ user }">{{ user.name }}</MyComp>` does **not** include `user` as a prop (it is bound by the slot directive).
- `Props()` on a component using `<MyComp v-slot="{ a, b }">` does not include `a` or `b` as props.
- `Props()` on a component using `<MyComp #item="{ name }">` does not include `name` as a prop.
- `Props()` on a component using `<MyComp v-slot="slotProps">{{ slotProps.x }}</MyComp>` does not include `slotProps` as a prop.
- Identifiers referenced *outside* the slot content (in sibling or ancestor elements) are not affected by slot-directive locals.
- All existing `Props()` tests continue to pass.
